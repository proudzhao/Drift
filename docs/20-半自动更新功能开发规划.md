# Drift · 半自动更新功能开发规划

本文规划 Drift 从“检查更新后跳转 GitHub 下载”升级为“用户确认后在应用内下载并安装更新”的开发路径。目标是减少用户每次更新都手动打开 GitHub、下载安装包、重新安装的成本，同时避免静默更新打断直播使用场景。

本文是开发规划文档，实际功能状态以代码、发布 workflow 和 `docs/当前功能文档.md` 为准。Tauri updater 的配置、签名和发布产物格式以实施时官方文档为准；截至 2026-06-23，Tauri v2 官方 updater 文档要求更新包带 updater 签名，并通过 `capabilities` 显式开放检查、下载和安装权限。

参考：

- Tauri v2 Updater 文档：`https://v2.tauri.app/plugin/updater/`
- Tauri GitHub Action：`https://github.com/tauri-apps/tauri-action`
- Drift 公开仓库工作流：`docs/06-Git公开仓库工作流.md`

---

## 一、目标

1. 在 Drift 设置页中支持检查到新版本后，用户点击“下载并安装”完成应用内更新。
2. 保留“启动时自动检查更新”配置，但只自动检查和提示，不自动下载、不自动安装。
3. 下载过程提供明确进度、错误信息和重试入口。
4. 安装前提示用户当前应用可能关闭或重启，尤其是 Windows 平台。
5. 改造 GitHub Releases 发布流程，使公开发布资产包含 Tauri updater 所需的更新包、`.sig` 签名和 `latest.json`。
6. 保持现有手动下载入口作为兜底路径，避免 updater manifest 或网络异常时用户无法更新。

---

## 二、非目标

- 不做静默自动下载。
- 不做静默自动安装。
- 不做后台强制重启。
- 不做增量更新或差分更新。
- 不自建动态更新服务器，第一期只使用 GitHub Releases 静态 `latest.json`。
- 不解决 Windows“未知发布者”或 macOS“无法验证/文件损坏”问题；这些仍属于系统代码签名、公证和证书问题。
- 不支持降级安装。
- 不支持 beta、stable 多通道更新；第一期只面向公开正式版本。
- 不把 updater 私钥提交到仓库。

---

## 三、重要前提

### 3.1 首次进入在线更新链路仍需手动安装一次

当前 v0.6.2 只有“检查更新并跳转下载”的能力，没有内置 updater 插件。因此已经安装 v0.6.2 的用户无法直接在线更新到第一个带半自动更新的版本。

第一版半自动更新发布后，用户仍需要手动下载安装一次，例如：

```text
v0.6.2 -> 手动下载安装 v0.6.3
v0.6.3 -> 之后可在应用内半自动更新到 v0.6.4+
```

这需要在 Release Notes 和 README 中明确说明。

### 3.2 updater 签名不等于系统代码签名

Tauri updater 签名用于证明更新包来自可信发布者，防止更新包被篡改。它不能替代：

- Windows 代码签名证书。
- macOS Developer ID 签名。
- macOS notarization 公证。

所以本功能能改善“后续更新流程”，不能消除首次安装时的系统安全提示。

### 3.3 public-main 发布 workflow 必须同步改造

Drift 的公开仓库发布 workflow 位于 `public-main` 分支的 `.github/workflows/release.yml`，不在 `main` 的私有文档分支中。实现阶段需要：

1. 在 `main` 完成应用源码和文档修改。
2. 按 `docs/06-Git公开仓库工作流.md` 同步 `drift/` 到 `public-main`。
3. 在 `public-main` 单独修改 release workflow。
4. 推送 `public-main -> origin/main`。

## 四、当前基础

当前 Drift 已具备：

- Tauri v2 + React + Vite 应用结构。
- `drift/src-tauri/src/update_check.rs`：Rust 端调用 GitHub Releases API 检查最新版本。

- `drift/src/components/control/AboutSettings.tsx`：关于页展示版本号、手动检查更新和“前往下载”按钮。
- `drift/src/components/control/ControlPanel.tsx`：控制面板顶部展示新版本提示条。
- `drift/src/types/config.ts` 和 `drift/src-tauri/src/app_config.rs`：已有 `update.checkOnStartup` 配置。
- `public-main` 的 `.github/workflows/release.yml`：当前发布 Windows x64 安装包和 macOS Apple Silicon / Intel DMG。

规划初始缺口：

- 未安装 `tauri-plugin-updater` 和 `@tauri-apps/plugin-updater`。
- 未配置 `bundle.createUpdaterArtifacts`。
- 未配置 updater `pubkey` 和 `endpoints`。
- 未在 `capabilities/default.json` 开放 `updater:default` 权限。
- Release 资产未明确包含 updater bundle、`.sig` 和 `latest.json`。
- 设置页没有下载、安装、进度和安装后重启交互。

## 五、总体方案

采用“官方 updater 插件 + GitHub Releases 静态 manifest + 用户确认安装”的方案。

核心流程：

```text
启动或手动点击检查更新
  -> @tauri-apps/plugin-updater check()
  -> 读取 GitHub Releases latest.json
  -> Tauri 校验版本号和签名信息
  -> 前端提示发现新版本
  -> 用户点击“下载并安装”
  -> updater 下载更新包并校验 .sig
  -> 安装完成
  -> 提示重启或说明 Windows 已由安装器接管退出
```

发布流程：

```text
生成 updater key pair
  -> public key 写入 tauri.conf.json
  -> private key 写入 GitHub Secrets
  -> release workflow 构建各平台安装包
  -> Tauri 生成 updater artifacts 和 .sig
  -> workflow 生成或上传 latest.json
  -> GitHub Release 发布为正式版本
```

第一期推荐保留旧 GitHub Release 页面入口：

- updater 可用时显示“下载并安装”。
- updater 检查失败时显示错误和“前往 GitHub 下载”。
- updater manifest 缺失或签名校验失败时不自动降级安装，只允许手动打开 GitHub。

## 六、用户体验设计

### 6.1 控制面板顶部提示

当启动检查发现新版本时：

```text
发现新版本 v0.6.4
当前版本 0.6.3，可在应用内下载并安装。
[下载并安装] [查看说明] [稍后]
```

行为：

- “下载并安装”：进入下载/安装流程。
- “查看说明”：打开 Release 页面。
- “稍后”：只关闭当前提示，不修改 `checkOnStartup`。

### 6.2 关于页

关于页保留当前布局，更新区域变为状态机：

```text
未检查
检查中
已是最新版本
发现新版本
下载中 38%
安装中
安装完成，重启后生效
检查失败 / 下载失败 / 安装失败
```

按钮策略：

- 未检查：`检查更新`
- 已是最新：`重新检查`
- 发现新版本：`下载并安装`、`前往 GitHub 下载`
- 下载中/安装中：按钮禁用，显示进度
- 失败：`重试`、`前往 GitHub 下载`
- 安装完成：`重启 Drift`

### 6.3 Windows 特殊提示

Tauri 官方文档说明 Windows 安装更新时，应用可能会因为安装器限制而自动退出。因此 Windows 上点击安装前需要提示：

```text
安装更新会关闭 Drift。请确认当前直播使用场景可以中断。
[确认安装] [取消]
```

macOS 上仍建议提示用户安装完成后重启，但不需要夸大为系统级风险提示。

## 七、技术设计

### 7.1 依赖

Rust：

```toml
tauri-plugin-updater = "2"
tauri-plugin-process = "2"
```

前端：

```json
"@tauri-apps/plugin-updater": "^2",
"@tauri-apps/plugin-process": "^2"
```

`tauri-plugin-process` 用于安装完成后从前端触发重启。若实施时确认 Tauri 核心 API 已能满足重启需求，可不引入该插件，但阶段验收必须覆盖“安装完成后用户能明确重启或退出重开”。

### 7.2 Tauri 配置

`drift/src-tauri/tauri.conf.json` 需要新增：

```json
{
  "bundle": {
    "createUpdaterArtifacts": true
  },
  "plugins": {
    "updater": {
      "pubkey": "这里填写 updater 公钥文件内容",
      "endpoints": [
        "https://github.com/proudzhao/Drift/releases/latest/download/latest.json"
      ]
    }
  }
}
```

约束：

- `pubkey` 是公钥内容，可以提交。
- 私钥只能保存在本地安全位置或 GitHub Secrets。
- 生产环境 endpoint 必须使用 HTTPS。
- endpoint 使用 `releases/latest/download/latest.json` 时，公开 Release 必须是 GitHub 能识别的 latest release。发布流程不能长期停留在 draft，也不能只发布不面向用户的 prerelease。

### 7.3 权限

`drift/src-tauri/capabilities/default.json` 增加：

```json
"updater:default",
"process:allow-restart"
```

如果实施时 process 插件权限名称与官方文档不同，以当前生成的 Tauri schema 为准，阶段 1 必须通过 `cargo check` 和 `npm run build` 验证。

### 7.4 前端状态模型

建议新增前端类型：

```ts
type AppUpdateStatus =
  | "idle"
  | "checking"
  | "available"
  | "not_available"
  | "downloading"
  | "installing"
  | "installed"
  | "error";

type AppUpdateState = {
  status: AppUpdateStatus;
  currentVersion: string;
  latestVersion: string;
  releaseUrl: string;
  notes: string;
  downloadedBytes: number;
  totalBytes?: number;
  error: string;
  checkedAt?: number;
};
```

前端 hook 建议从现有 `useUpdateNotice.ts` 演进为：

```text
drift/src/hooks/control/useAppUpdate.ts
```

职责：

- 读取当前版本。
- 根据 `checkOnStartup` 控制启动检查。
- 调用 updater `check()`。
- 保存 `Update` 对象或封装安装动作。
- 管理下载进度。
- 处理错误和重试。
- 向控制面板和关于页暴露同一份状态。

### 7.5 旧更新检查逻辑处理

阶段实现完成后，旧的 `update_check.rs` 不应继续和 updater 并行检查同一件事，否则会出现两套结果和两套错误路径。

推荐处理：

- 关于页版本号改用 `@tauri-apps/api/app` 或保留一个极小的 `get_app_version` command。
- 移除或停用 `check_update`、`get_cached_update_result` 和 `update-available` 事件。
- 如果保留旧 GitHub API 检查，只作为 updater 初始化失败时的手动 fallback，不参与启动自动检查。

## 九、阶段规划

### 阶段 0：发布产物与 updater key 验证

目标：先确认 Tauri updater 在 Drift 当前打包矩阵中能生成可用产物，避免 UI 实现后卡在发布链路。

建议改动：

- 在本地或临时分支生成 updater key pair。
- 记录公钥、私钥保存位置和 GitHub Secrets 名称，不记录私钥内容。
- 临时配置 `createUpdaterArtifacts` 和 updater `pubkey`。
- 本地执行至少一个平台的 release build。
- 检查构建产物中是否出现 updater bundle 和 `.sig`。
- 在 `public-main` 的 release workflow 上验证 tauri-action 是否自动上传 `latest.json`；如果不能，明确后续要增加 manifest 生成脚本。

验收：

- 明确 Drift 使用的 updater public key。
- 明确 GitHub Secrets 名称，例如 `TAURI_SIGNING_PRIVATE_KEY` 和 `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`。
- 明确每个平台发布时需要上传哪些 updater 资产。
- 明确 `latest.json` 是由 tauri-action 自动生成，还是由 Drift workflow 自己生成。
- 没有私钥内容进入 Git。

阶段 0 实测记录（2026-06-23）：

- 本地 Tauri CLI 版本：`tauri-cli 2.11.2`。
- `npm run tauri -- signer generate --ci` 可生成 updater key pair。
- 阶段 0 临时 key pair 已生成在 `/private/tmp/drift-updater-phase0.key` 和 `/private/tmp/drift-updater-phase0.key.pub`，只用于本地验证，不作为正式发布 key。
- 临时私钥权限已收紧为 `600`，私钥内容未写入 Git。
- 临时公钥内容如下，后续正式接入阶段需要换成正式公钥：

```text
dW50cnVzdGVkIGNvbW1lbnQ6IG1pbmlzaWduIHB1YmxpYyBrZXk6IDQxRUNFQUVBOTBCOEE3NDgKUldSSXA3aVE2dXJzUVl3aXZDMk9qSVFjaHpyeGJFTDg3NzRpRjN1dWpUVU41MXhJWWdLajFDcmEK
```

- 本地最小构建命令使用 `--config` 临时合并 `createUpdaterArtifacts: true`、updater `pubkey` 和 GitHub `latest.json` endpoint，未修改主配置文件。
- `--no-sign` 会同时跳过 updater 签名，只能生成 `drift.app.tar.gz`，不能生成 `.sig`，不满足 updater 验收。
- Tauri build 阶段需要 `TAURI_SIGNING_PRIVATE_KEY` 环境变量包含私钥内容；仅设置 `TAURI_SIGNING_PRIVATE_KEY_PATH` 会失败并提示没有 private key。
- 使用临时私钥内容设置 `TAURI_SIGNING_PRIVATE_KEY` 后，macOS `.app` 最小构建通过，并生成：

```text
drift/src-tauri/target/release/bundle/macos/drift.app.tar.gz
drift/src-tauri/target/release/bundle/macos/drift.app.tar.gz.sig
```

- `drift.app.tar.gz` 大小约 `6.8M`，`drift.app.tar.gz.sig` 大小约 `404B`，签名文件为可内联进 `latest.json` 的文本内容。
- 当前 `public-main` 的 `.github/workflows/release.yml` 使用 `tauri-apps/tauri-action@v0`，发布为 `releaseDraft: true` 和 `prerelease: true`，且未注入 updater signing secret。
- `tauri-action` 当前文档显示：如果 updater 已配置，`uploadUpdaterJson` 默认 `true`，`uploadUpdaterSignatures` 默认 `true`；阶段 4 可优先升级到 `tauri-apps/tauri-action@v1` 并依赖 action 自动生成 `latest.json`，只有自动生成不满足 Drift 需求时再自建 manifest 脚本。
- 正式发布建议使用 GitHub Secrets：
  - `TAURI_SIGNING_PRIVATE_KEY`：正式 updater 私钥内容。
  - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`：正式 updater 私钥密码。
  - 不建议只依赖 `TAURI_SIGNING_PRIVATE_KEY_PATH`。

阶段 0 结论：

- Tauri updater 产物链路在 Drift 当前 macOS 本地环境中可行。
- 下一阶段可以进入 updater 依赖、配置和权限接入。
- GitHub 上暂时不需要手动操作；等阶段 4 改造 release workflow 前，需要你在公开仓库 Settings -> Secrets and variables -> Actions 中添加正式 updater signing secrets。

### 阶段 4：GitHub Release workflow 改造

目标：公开发布流程能稳定生成半自动更新所需资产。

建议改动：

- 在 `public-main` 修改 `.github/workflows/release.yml`。
- 为 tauri-action 注入 updater signing secret。
- 确认 release build 生成 updater artifacts。
- 确认 Release 中包含 `latest.json`。
- 如果 tauri-action 没有生成符合 Drift 需要的 `latest.json`，新增脚本在 workflow 中根据 release assets 和 `.sig` 生成静态 manifest。
- 将 release 从长期 `draft: true` / `prerelease: true` 的预览流程，调整为支持正式发布的参数。

验收：

- 一次测试 Release 中包含 Windows x64、macOS Apple Silicon、macOS Intel 对应的更新资产。
- `latest.json` 可通过 `https://github.com/proudzhao/Drift/releases/latest/download/latest.json` 访问。
- `latest.json` 中 `version` 是合法 SemVer，允许带 `v` 前缀。
- `platforms` 中包含 `windows-x86_64`、`darwin-aarch64`、`darwin-x86_64`。
- 每个平台条目都有可下载 `url` 和内联 `signature` 内容。
- 私钥只存在于 GitHub Secrets，不出现在 workflow 日志或仓库文件中。

阶段 4 实施记录（2026-06-23）：

- 已在 `public-main` 改造 `.github/workflows/release.yml`。
- 发布 workflow 保留当前可用的 `tauri-apps/tauri-action@v0`；`tauri-action@v1` 在验证时没有可用 tag，因此不绑定不存在的版本。
- workflow dispatch 新增 `draft`、`prerelease` 和 `release_body` 参数，默认发布为非 draft、非 prerelease 的正式 Release。
- release build 注入 `TAURI_SIGNING_PRIVATE_KEY` 和可选的 `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`；打包前检查私钥存在，无密码 key 允许 password 为空。
- 显式启用 `includeUpdaterJson: true`，并设置 `updaterJsonPreferNsis: true`，让 Windows updater manifest 优先使用 NSIS 安装包。
- 构建后扫描并上传 `.sig` 文件，手动上传的签名资产会加平台前缀避免 macOS 双架构覆盖；如果没有生成签名文件，workflow 直接失败。
- 新增 `validate-updater-manifest` job：下载 Release 中的 `latest.json`，校验 `version`、`windows-x86_64`、`darwin-aarch64`、`darwin-x86_64`，并确认每个平台都有 `url` 和内联 `signature`；同时检查三个平台的 `.sig` 资产已上传。
- 当发布不是 draft 且不是 prerelease 时，额外验证 `https://github.com/proudzhao/Drift/releases/latest/download/latest.json` 可访问。
- 普通 Windows 构建 workflow 改为 `--no-sign`，避免非发布构建要求 updater 私钥。

GitHub 手动配置要求：

- 在公开仓库 Settings -> Secrets and variables -> Actions 中添加 `TAURI_SIGNING_PRIVATE_KEY`。
- 如果 key pair 设置了密码，在公开仓库 Settings -> Secrets and variables -> Actions 中添加 `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`；无密码 key 可以不添加或留空。
- 这些 secret 必须与 `drift/src-tauri/tauri.conf.json` 中当前 updater `pubkey` 对应；如果重新生成正式 key pair，需要同步替换 `pubkey`。

## 十、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| `latest.json` 未正确生成 | 应用无法发现新版本 | 阶段 0 先验证 tauri-action 行为；必要时 workflow 自建 manifest |
| 私钥泄漏 | 更新链路可信性失效 | 私钥只放 GitHub Secrets；泄漏后更换 key pair 并发布新版本 |
| Release 一直是 draft 或 prerelease | `releases/latest` endpoint 可能拿不到目标版本 | 正式版本发布时使用非 draft，并明确 prerelease 策略 |
| Windows 安装时应用自动退出 | 用户直播中断 | 安装前确认提示；不做静默安装 |
| macOS 未签名或未公证 | 首次安装仍有系统阻拦 | 文档说明边界；后续签名公证作为独立发布工程 |
| updater 下载失败 | 用户无法应用内更新 | 保留 GitHub 下载入口 |
| 旧检查更新逻辑和新 updater 并行 | UI 重复提示或状态不一致 | 阶段 3 统一到 `useAppUpdate` |
